## Workflow par feature (boucle, mode nuit)

Pour chaque ligne non cochée de `.hermes-queue.md`, dans l'ordre :

### 1. Préparation
- `git checkout main && git pull --ff-only`
- **NE PAS** créer la branche manuellement — `/speckit.specify` crée la
  branche feature automatiquement (Spec Kit gère ça via son hook).
- Logger dans `.hermes/logs/orchestration.log` :
  `<ISO timestamp> | NN-slug | START`

### 2. Spec Kit — Phase préparation

Délègue à **une session Claude Code A** (réutilisable pour `specify` →
`clarify` → `plan` → `tasks` → `analyze`) :

1. `/speckit.specify "$(cat docs_et_brouillons/features/NN-<slug>.md)"`
   — Spec Kit crée la branche, le `spec.md` et bascule dessus.

2. `/speckit.clarify`
   — **Mode autonome (l'utilisateur dort)** :
     - Pour CHAQUE question posée : choisir l'option **`Recommended`** /
       `(recommended)` / la valeur par défaut explicite.
     - S'il n'y a aucune recommandation explicite (cas rare) : répondre selon
       la logique du projet en privilégiant **dans cet ordre** :
       1. l'invariant Module 0 le plus restrictif (sourcing, RLS, audit log,
          versioning, money typé, multi-tenant) ;
       2. la stack imposée par `00-INDEX.md` (Nuxt 4, FastAPI, PostgreSQL+
          pgvector, Voyage AI, minimax-m2.7) ;
       3. le choix le plus simple et le plus testable.
     - Logger chaque réponse non-recommandée dans `.hermes/logs/clarify-NN.log`
       avec la justification, pour revue au matin.
     - **Ne jamais** s'arrêter sur une clarification — toujours décider.

3. `/speckit.plan`

4. `/speckit.tasks`

5. `/speckit.analyze`
   — Si `CRITICAL` ou `HIGH` : demander à la même session Claude Code A de
     corriger (max 2 itérations), puis re-run `analyze`. Si toujours bloqué
     après 2 itérations → STOP (cf. Stop conditions).

### 3. Spec Kit — Phase implémentation (session FRAÎCHE)

**Impératif** : `/speckit.implement` DOIT s'exécuter dans une **session
Claude Code B** distincte de A (contexte propre).

Au choix selon le skill :
- Soit ouvrir une nouvelle invocation `claude-code` (préféré).
- Soit, dans la session existante, exécuter `/clear` AVANT `/speckit.implement`.

Puis :

6. `/speckit.implement`
   — Suit TDD (tests d'abord, couverture ≥ 80 %).
   — Respecte les invariants Module 0.
   — Ne JAMAIS skipper de tests automatisés (`pytest`, `vitest`).
   — Les **tests manuels** documentés dans `tasks.md` (E2E navigateur, vérif
     UI à l'œil, etc.) sont **laissés à l'utilisateur** — ne pas tenter de
     les jouer, juste les noter dans `.hermes/logs/manual-tests-NN.md`.

### 4. Boucle de correction post-implement (MÊME session B)

Après `/speckit.implement`, exécute la suite de tests **automatisés** :
- Backend : `cd backend && source .venv/bin/activate && pytest --cov`
- Frontend : `cd frontend && pnpm test` (si tests configurés)
- Lint : `ruff check backend/` et `pnpm lint` côté front si dispo

Si tests **rouges** :
- Renvoyer les erreurs à la **même session Claude Code B** (PAS de `/clear`,
  PAS de session fraîche — on garde tout le contexte d'implémentation).
- Lui demander de corriger.
- Re-run les tests.
- **Boucler jusqu'à vert** — pas de limite stricte tant que la session
  progresse. Tracker dans `.hermes/logs/fixloop-NN.log` :
  `<timestamp> | iteration N | tests failed: X → Y`.
- **Stop seulement si** :
  - 5 itérations consécutives sans réduction du nombre de tests rouges
    (= boucle qui ne progresse plus) ;
  - OU régression sur des tests d'une feature **précédente** déjà cochée.

### 5. Commit & PR
- `git add -A`
- `git commit -m "feat(NN): <description courte tirée de spec.md>"`
- `git push -u origin <branche-créée-par-speckit>`
- `gh pr create --base main --fill --label "auto-generated"` (gh est installé
  et authentifié comme `angenor`).
- Récupérer l'URL de la PR.

### 6. Mise à jour queue + log
- Cocher la case `[ ]` → `[x]` dans `.hermes-queue.md` pour la feature NN.
- Append dans `.hermes/logs/orchestration.log` :
  `<ISO timestamp> | NN-slug | DONE | PR=<url> | manual-tests=<chemin du fichier>`
- Passer à la feature suivante **sans interruption**.

## Stop conditions (réveiller l'utilisateur via message clair)

Réveille l'utilisateur **uniquement** dans ces cas :

1. Le repo est dans un état git incohérent (conflit non résolu, HEAD détaché).
2. Une commande Spec Kit échoue de manière non récupérable (binaire `claude`
   absent, ou erreur API persistante > 10 min).
3. La boucle de correction post-implement stagne (5 itérations sans progrès).
4. Régression détectée sur une feature précédemment validée.
5. `gh pr create` échoue (auth expirée, rate limit, etc.).
6. Une migration Alembic supprimerait des données existantes.
7. Une dépendance bloque toute la file (ex : F02 échoue → F03..F05 dépendent
   d'elle, donc on ne peut pas continuer la queue).

Format du message de réveil (laissé en tête de session) :

```
🚨 BLOCAGE — feature NN-<slug>
Étape : <speckit.X | tests | git | gh>
Erreur : <résumé 3 lignes max>
Action requise : <ce que doit faire l'utilisateur>
État repo : branche=<X>, dernier commit=<sha>, tests=<rouge/vert>
Logs détaillés : .hermes/logs/<fichier>
```

## Garde-fous (non négociables)

1. **Ne jamais** committer sur `main` directement.
2. **Ne jamais** `--no-verify`, `--force-push`, `git reset --hard`.
3. **Ne jamais** hardcoder de secret (utiliser `backend/.env.example`).
4. **Ne jamais** supprimer de fichier hors du périmètre de la feature courante.
5. **Ne jamais** réveiller l'utilisateur pour une question de clarification
   normale — c'est à toi de décider (voir § Mode autonome ci-dessus).
6. **Toujours** créer un commit *avant* de basculer vers une autre feature.
7. **Toujours** utiliser le format conventional commits : `feat(NN): ...`,
   `fix(NN): ...`, `test(NN): ...`.

## Stack & invariants techniques (rappel)

- **Backend** : FastAPI + Python 3.11+ + venv local, Alembic
- **Frontend** : Nuxt 4 + Pinia + TailwindCSS v4, `pnpm`
- **DB** : PostgreSQL 16 + pgvector (seul service dockerisé)
- **LLM** : minimax-m2.7 via OpenRouter (`LLM_BASE_URL` + `LLM_MODEL`)
- **Embeddings** : Voyage AI (`voyage-3.5`, 1024 dims)
- **Hébergement cible** : Europe / Afrique de l'Ouest (pas USA, RGPD)

Invariants Module 0 (à respecter dans CHAQUE feature) :
- Sourcing obligatoire de tout chiffre/critère/seuil.
- Multi-tenant strict (RLS PostgreSQL + `account_id`).
- Audit log append-only de toute mutation.
- Versioning des référentiels et critères.
- Money typé `{amount, currency}` avec peg FCFA-EUR fixe 655,957.
- UI bottom sheet pour les composants interactifs.
- Plateforme fermée : seuls `PME` et `Admin` existent.
