## Workflow par feature (boucle)

Pour chaque ligne non cochée de `.hermes-queue.md`, dans l'ordre :

### 1. Préparation
- `git checkout main && git pull --ff-only`
- `git checkout -b feat/NN-<slug>` (slug = nom de fichier sans le numéro ni `.md`)
- Logger dans `.hermes/logs/orchestration.log` : `START NN-slug`

### 2. Spec Kit (déléguer à `claude-code`)

Délègue à `claude-code` la séquence suivante, **dans le repo** :

1. `/speckit.specify "$(cat docs_et_brouillons/features/NN-<slug>.md)"`
2. `/speckit.clarify`
   - Si la sortie contient `[NEEDS CLARIFICATION]` non résolus → **STOP** et reporte les questions à l'utilisateur.
3. `/speckit.plan`
4. `/speckit.tasks`
5. `/speckit.analyze`
   - Si l'analyse signale des incohérences `CRITICAL` ou `HIGH` → corriger avant `implement`.
6. `/speckit.implement`
   - Doit suivre TDD (tests avant code, couverture ≥ 80 %).
   - Doit respecter les **invariants Module 0** (sourcing, multi-tenant, audit log, versioning, money typé, mapping ESG, RLS).

### 3. Vérifications
- Lancer la suite de tests backend (pytest) et frontend (vitest si applicable).
- Lancer le linter (`ruff` côté backend, `eslint` côté front si configurés).
- Si rouge → demander à `claude-code` de corriger. Max 2 itérations, sinon STOP.

### 4. Commit & PR
- `git add -A && git commit -m "feat(NN): <description courte>"`
- `git push -u origin feat/NN-<slug>`
- Si `gh` CLI est disponible : `gh pr create --base main --fill`.
  - Si `gh` n'est pas installé : afficher l'URL `https://github.com/angenor/esg_mefali_v2/compare/feat/NN-<slug>?expand=1` à l'utilisateur.

### 5. Mise à jour queue
- Cocher la case correspondante dans `.hermes-queue.md`.
- Logger dans `.hermes/logs/orchestration.log` : `DONE NN-slug | PR=<url ou compare-url>`.

## Garde-fous (non négociables)

1. **Ne jamais** committer/pusher sur `main`.
2. **Ne jamais** supprimer ou écraser des fichiers existants sans demander.
3. **Ne jamais** skipper les tests (`--no-verify`, `pytest -x` non autorisé sans raison).
4. **Ne jamais** hardcoder de secret. Utiliser `backend/.env.example` comme référence.
5. **Stop conditions** :
   - 2 échecs successifs sur la même étape.
   - Un test casse une feature précédente (régression).
   - Spec Kit demande une clarification non résolvable depuis le markdown.
   - Conflit git non trivial.
6. **Budget** : si plus de 30 min sur une seule feature, pause et reporte un état d'avancement.
7. **Mémoire personnelle** : utiliser `update_personal_notes` pour mémoriser les choix d'architecture importants entre features.

## Stack & invariants techniques (rappel)

- **Backend** : FastAPI + Python 3.11+ + venv local, Alembic
- **Frontend** : Nuxt 4 + Pinia + TailwindCSS v4, `pnpm`
- **DB** : PostgreSQL 16 + pgvector (seul service dockerisé)
- **LLM** : minimax-m2.7 via OpenRouter (interchangeable via `LLM_BASE_URL`)
- **Embeddings** : Voyage AI (`voyage-3.5`)
- **Hébergement cible** : Europe ou Afrique de l'Ouest (pas USA, RGPD)

Invariants Module 0 :
- Sourcing obligatoire de tout chiffre/critère/seuil.
- Multi-tenant strict (RLS PostgreSQL + `account_id`).
- Audit log append-only de toute mutation.
- Versioning des référentiels et critères.
- Money typé `{amount, currency}` avec peg FCFA-EUR fixe 655,957.
- UI bottom sheet pour les composants interactifs.
