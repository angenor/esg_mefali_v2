# Orchestrateur Claude Code — ESG MEFALI v2 (mode parallèle)

> Document à charger dans une session Claude Code fraîche.
> **Mode** : parallèle dynamique. Plan max → jusqu'à 6 sous-agents en simultané.
> L'utilisateur est absent (courses) — autonomie totale jusqu'à épuisement
> de la queue ou stop conditions.

---

## Rôle

Tu es **toi-même Claude Code**, ouvert dans
`/Users/mac/Documents/projets/2025/esg_mefali_v2`. Tu es **l'orchestrateur
parallèle**. Tu n'écris JAMAIS de code toi-même — tu délègues à des
sous-agents `Task` (subagent_type = `general-purpose`), chacun fraîs.

Boucle principale (jusqu'à `done` = 35 ou stop) :

1. **Lire l'état** : `.cc-deps.json` (deps, done, in_progress, blocked).
2. **Calculer ready** : `ready = {f : f ∉ done ∪ in_progress ∪ blocked
   ET deps(f) ⊆ done}`.
3. **Lancer en // jusqu'à `_max_parallel`** : pour chaque feature `ready`
   non encore lancée, démarrer un sous-agent Phase A en `Task` parallèle.
4. **Attendre les rapports** : récupérer JSON, gérer les transitions
   (Phase A → Phase B → commit/push/PR/merge → done), maintenir
   `.cc-deps.json` à jour à chaque transition.
5. **Boucler** : recalculer `ready` (de nouvelles features peuvent l'être
   après chaque merge), relancer.

## Contraintes parallèles — règles d'or

### Plafond et synchronisation

- **`_max_parallel` = 6** (lu dans `.cc-deps.json`). Plan max permet 10
  mais on garde de la marge pour : (a) ne pas saturer le contexte de
  l'orchestrateur, (b) sérialiser proprement les merges, (c) absorber les
  pics de boucles de correction.
- **Implementations en //, merges en série** : tu peux avoir 6 features
  qui codent en parallèle, mais tu MERGES les PR une par une dans l'ordre
  où elles deviennent prêtes. Cela évite tout conflit de merge.
- **Squash + delete branch** sur chaque merge : `gh pr merge <num>
  --squash --delete-branch`.

### Zones interdites en parallèle (un seul écrivain à la fois)

Si la feature A et la feature B touchent l'un de ces fichiers, **elles
doivent s'exécuter en série** (pas dans la même vague parallèle) :

- `backend/alembic/versions/` — pour éviter les conflits de
  `down_revision`. **Règle** : une seule feature avec migration alembic
  en flight à la fois.
- `backend/app/main.py` — registration des routers (édition de la même
  zone du fichier).
- `backend/app/core/config.py` — ajout de variables d'environnement.
- `backend/conftest.py` (racine) ou fixtures globales.
- `backend/pyproject.toml`, `backend/uv.lock`.
- `frontend/nuxt.config.ts`, `frontend/app.vue`, layouts globaux,
  composants de menu/navigation.
- `frontend/package.json`, `frontend/pnpm-lock.yaml`.
- `docker-compose.yml`.
- `.cc-deps.json` (toi seul l'édites).
- `.cc-queue.md` (toi seul l'édites).

**Comment faire respecter** : dans le prompt du sous-agent, injecte la
liste des fichiers actuellement détenus par d'autres sous-agents en
flight. S'il a besoin d'y toucher, il doit signaler `zone_conflict` au
lieu d'écrire — tu re-coordonnes.

### Détection de conflit dur

Si `gh pr merge` échoue pour conflit avec une PR déjà mergée :
1. `git fetch origin && git checkout <branche-pr>`.
2. Lance un mini sous-agent Task `general-purpose` : "rebase sur
   origin/main et résous les conflits sans casser les tests".
3. Re-push, re-tente `gh pr merge`.
4. Si 2 tentatives échouent → marquer `blocked` dans `.cc-deps.json`,
   passer à la suivante.

## Workflow par feature

### 1. Préparation (toi)

- `git checkout main && git pull --ff-only`.
- Marquer la feature `in_progress` dans `.cc-deps.json`.
- Logger dans `.cc-runtime/logs/orchestration.log` :
  `<ISO> | NN-slug | START | parallel_count=<n>`.

### 2. Phase A — Spec Kit préparation (sous-agent FRAIS)

Lance un `Task` (general-purpose). Prompt :

```

  RLS PostgreSQL avec account_id, audit log append-only, versioning des
  référentiels, money typé {amount, currency} avec peg FCFA-EUR 655.957,
  UI bottom sheet, plateforme fermée (PME et Admin uniquement).

ZONES INTERDITES (autres sous-agents y travaillent — si tu DOIS y toucher,
signale-le dans ton rapport au lieu de modifier) :

RAPPORT JSON FINAL (ton dernier message au parent) :
```

### 3. Sérialisation alembic (toi, avant Phase B)

Si `alembic_migration_planned: true` ET une autre feature avec migration
est déjà en Phase B → **mettre cette feature en attente** (statut
`waiting_alembic`). Reprendre dès que l'autre a mergé.

Lance un second `Task` (general-purpose). Prompt :

### 5. Commit, push, PR, merge (toi, série)

Si `ready_for_pr: true` :

```bash
git checkout <branche>
git add -A
git commit -m "feat(NN): <titre court tiré de spec.md>"
git push -u origin <branche>
gh pr create --base main --fill --label "auto-generated"
# Capture URL et num de la PR
gh pr merge <num> --squash --delete-branch
git checkout main && git pull --ff-only
```

Si `gh pr merge` échoue (conflit) → procédure § Détection de conflit dur.

### 6. Mise à jour état (toi)

- Édite `.cc-deps.json` :
  - Retirer NN de `in_progress`.
  - Ajouter NN à `done`.
- Édite `.cc-queue.md` : `[ ]` → `[x]` pour la feature NN.
- Append dans `.cc-runtime/logs/orchestration.log` :
  ```
  <ISO> | NN-slug | DONE | PR=<url> | merged | manual-tests=...
  ```

### 7. Recalcul ready et lancement nouvelles vagues

Après chaque merge :
1. Recalculer `ready`.
2. Si `len(in_progress) < _max_parallel` ET il existe des features
   `ready` non lancées → lancer immédiatement de nouvelles Phases A.

Tu vises `len(in_progress)` proche de `_max_parallel` tant qu'il reste
des features ready.

## Stop conditions

Mets une feature en `blocked` (et passe à la suivante) si :

1. Phase A reporte `ready_for_implement: false` après 2 retries.
2. Phase B reporte `tests_status: stuck` ou `regression` ou
   `zone_conflict` non résolu.
3. `gh pr create` échoue après 1 retry post-60 s.
4. `gh pr merge` échoue avec conflit non résoluble en 2 tentatives.
5. Migration Alembic supprimerait des données existantes.

**Continue toujours** avec les autres features. N'arrête la queue qu'en
cas de :

- État git incohérent global (HEAD détaché, repo corrompu).
- Toutes les features restantes sont `blocked`.
- API Anthropic / gh down de manière persistante (> 10 min).

## Garde-fous (non négociables)

1. Jamais de commit sur `main` directement.
2. Jamais de `--no-verify`, `--force-push`, `git reset --hard`.
3. Jamais de secret hardcodé (utiliser `backend/.env.example`).
4. Jamais de suppression hors périmètre.
5. Jamais déranger l'utilisateur — il est en courses.
6. Toujours commit avant merge.
7. Toujours format conventional commits : `feat(NN): ...`.
8. Toujours utiliser `Task` (sous-agent fraîs) pour Phase A et Phase B.
9. Toujours sérialiser les migrations Alembic.
10. Toujours mettre à jour `.cc-deps.json` après chaque transition.

- Multi-tenant strict (RLS PostgreSQL + `account_id`).
- Audit log append-only.
- Versioning des référentiels et critères.
- Money typé `{amount, currency}` avec peg FCFA-EUR 655,957.
- UI bottom sheet pour les composants interactifs.
- Plateforme fermée : seuls `PME` et `Admin`.
