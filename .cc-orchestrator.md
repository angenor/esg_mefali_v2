# Orchestrateur Claude Code — ESG MEFALI v2 (mode nuit)

> Document à charger dans une session Claude Code fraîche, qui pilote
> directement le pipeline Spec Kit sur les 35 features.
> **Mode** : nuit / autonomie totale. L'utilisateur dort.

---

## Rôle

Tu es **toi-même Claude Code**, ouvert dans le repo
`/Users/mac/Documents/projets/2025/esg_mefali_v2`.

Tu es **l'orchestrateur**. Tu :
- lis la queue `.cc-queue.md` ;
- pour chaque feature non cochée, **lances un sous-agent fraîs** via le tool
  `Task` (subagent_type = `general-purpose`) qui exécute la feature ;
- récupères son rapport, commits/push/PR, coches la queue, passes à la
  suivante ;
- ne lances **jamais** de slash command Spec Kit toi-même — tu les délègues
  au sous-agent qui aura un contexte propre.

L'isolation par sous-agent garantit qu'aucune feature ne pollue le contexte
de la suivante (équivalent à une session Claude Code fraîche par feature).

## Workflow par feature

Pour chaque case `[ ]` non cochée, dans l'ordre du fichier :

### 1. Préparation (toi, l'orchestrateur)

```bash
git checkout main && git pull --ff-only
```

Logger dans `.cc-runtime/logs/orchestration.log` :
```
<ISO> | NN-slug | START
```

### 2. Phase A — Spec Kit préparation (sous-agent FRAIS)

Lance UN sous-agent via le tool `Task` (subagent_type = `general-purpose`).
Le sous-agent a un contexte vierge et exécute Spec Kit. Prompt à lui passer :

```
Tu es Claude Code dans le repo /Users/mac/Documents/projets/2025/esg_mefali_v2.

CONTEXTE :
- Feature à implémenter : NN-<slug>
- Description : contenu intégral de docs_et_brouillons/features/NN-<slug>.md
  (lis ce fichier en premier).
- Stack imposée par 00-INDEX.md : FastAPI + Nuxt 4 + PostgreSQL+pgvector +
  Voyage AI + minimax-m2.7 via OpenRouter. Hébergement Europe / Afrique
  de l'Ouest. Backend Python 3.11+ avec venv local. Pgvector dockerisé.
- Invariants Module 0 NON NÉGOCIABLES : sourcing obligatoire, multi-tenant
  RLS PostgreSQL avec account_id, audit log append-only, versioning des
  référentiels, money typé {amount, currency} avec peg FCFA-EUR 655.957,
  UI bottom sheet, plateforme fermée (PME et Admin uniquement).

TÂCHE — exécute en séquence dans CETTE session, sans /clear :

1. Lis docs_et_brouillons/features/NN-<slug>.md.

2. Invoque le skill speckit-specify avec le contenu intégral de ce
   markdown comme description. Cela créera la branche feature et le
   fichier specs/<dossier>/spec.md.

3. Invoque le skill speckit-clarify.
   - Pour CHAQUE question : choisir l'option marquée "Recommended" /
     "(recommended)" / valeur par défaut explicite.
   - Si pas de recommandation : décider selon (1) invariant Module 0 le
     plus restrictif, (2) stack imposée, (3) choix le plus simple/testable.
   - Logger chaque réponse non-recommandée dans
     .cc-runtime/logs/clarify-NN.log avec justification.
   - NE JAMAIS demander à l'humain — décide toujours.

4. Invoque speckit-plan.

5. Invoque speckit-tasks.

6. Invoque speckit-analyze.
   - Si CRITICAL ou HIGH : corrige (max 2 itérations) puis re-run.
   - Si toujours bloqué : reporte STOP au parent avec détails.

LIVRABLES (à reporter au parent en JSON dans ton message final) :
{
  "feature": "NN-slug",
  "branch": "<nom branche créée par speckit-specify>",
  "spec_path": "<chemin vers spec.md>",
  "plan_path": "<chemin vers plan.md>",
  "tasks_path": "<chemin vers tasks.md>",
  "analyze_status": "ok|critical|high|fixed",
  "clarify_decisions": [<liste des décisions non-recommandées prises>],
  "ready_for_implement": true|false
}

CONTRAINTES :
- Ne commit pas, ne push pas — ton parent s'en chargera après l'implement.
- Ne touche pas à la queue `.cc-queue.md` — c'est le boulot du parent.
- Reste sur la branche feature créée par speckit-specify.
```

### 3. Phase B — Implémentation + correction (sous-agent FRAIS distinct)

Une fois la phase A `ready_for_implement: true`, lance un **second**
sous-agent (Task, general-purpose) avec contexte propre :

```
Tu es Claude Code dans le repo /Users/mac/Documents/projets/2025/esg_mefali_v2.

CONTEXTE :
- Feature : NN-<slug>
- Branche feature : <nom de la branche, doit déjà être checkout>
- Phase A a déjà produit : spec.md, plan.md, tasks.md (relis-les si besoin).
- Stack et invariants : voir docs_et_brouillons/features/00-INDEX.md
  (lis-le si nécessaire pour les rappels Module 0).

TÂCHE :

1. Vérifie que tu es bien sur la branche feature (git branch --show-current).
2. Invoque le skill speckit-implement.
   - TDD obligatoire : tests d'abord, code ensuite. Couverture ≥ 80 %.
   - Respecte STRICTEMENT les invariants Module 0.
   - Ne JAMAIS skipper de test automatisé (pas de `pytest -x` sans raison,
     pas de `--no-verify`).
   - Tests MANUELS de tasks.md (vérification UI à l'œil, E2E navigateur
     non automatisable) : NE PAS les jouer, les lister dans
     .cc-runtime/logs/manual-tests-NN.md pour que l'utilisateur les fasse
     à son réveil.

3. Une fois speckit-implement terminé, exécute la suite de tests automatisés :
   - Backend : `cd backend && source .venv/bin/activate && pytest --cov`
     (ou équivalent selon F01).
   - Frontend : `cd frontend && pnpm test` si configuré.
   - Lint : `ruff check backend/` et `pnpm lint` si dispo.

4. Si tests rouges :
   - Corrige les erreurs DANS CETTE MÊME SESSION (pas de /clear).
   - Re-run tests.
   - Boucle jusqu'au vert. Tracker dans .cc-runtime/logs/fixloop-NN.log :
     `<ts> | iteration N | failed: X → Y`.
   - STOP seulement si :
     • 5 itérations consécutives sans réduction du nombre de tests rouges, OU
     • régression sur une feature précédente déjà cochée.

LIVRABLES (rapport JSON au parent) :
{
  "feature": "NN-slug",
  "branch": "<branche>",
  "tests_status": "green|stuck|regression",
  "coverage": <float ou null>,
  "files_changed": [<liste>],
  "manual_tests_log": ".cc-runtime/logs/manual-tests-NN.md",
  "fixloop_iterations": <int>,
  "ready_for_pr": true|false,
  "blocker": "<description si stuck>"
}

CONTRAINTES :
- Ne commit pas, ne push pas — ton parent fera commit + push + PR.
- Ne touche pas à .cc-queue.md.
```

### 4. Commit, push, PR (toi, l'orchestrateur)

Si `ready_for_pr: true` :

```bash
git add -A
git commit -m "feat(NN): <description courte tirée de spec.md>"
git push -u origin <branche>
gh pr create --base main --fill --label "auto-generated"
```

Capture l'URL de la PR. Si une étape échoue (commit vide, conflict, gh
rate limit) → STOP via stop conditions.

### 5. Mise à jour queue + log (toi)

- Édite `.cc-queue.md` : remplacer `[ ] NN — ...` par `[x] NN — ...`.
- Append dans `.cc-runtime/logs/orchestration.log` :
  ```
  <ISO> | NN-slug | DONE | PR=<url> | manual-tests=.cc-runtime/logs/manual-tests-NN.md
  ```
- Passe à la feature suivante **sans pause**.

## Stop conditions (réveiller l'utilisateur)

Réveille l'utilisateur **uniquement** dans ces cas (laisse un message en tête
de session, n'arrête pas la queue si une feature suivante est indépendante) :

1. État git incohérent (conflit, HEAD détaché).
2. Phase A : sous-agent reporte `ready_for_implement: false` après 2 retries.
3. Phase B : sous-agent reporte `tests_status: stuck` ou `regression`.
4. `gh pr create` échoue (auth, rate limit) — réessaie une fois après 60 s,
   puis stop.
5. Une migration Alembic supprimerait des données existantes.
6. Si la feature actuelle bloque toutes les suivantes (dépendance critique).

Format de réveil :

```
🚨 BLOCAGE — feature NN-slug
Étape : <phase A | phase B | commit | push | pr>
Erreur : <résumé 3 lignes max>
Action requise : <ce que doit faire l'utilisateur>
État repo : branche=<X>, dernier commit=<sha>, tests=<rouge/vert>
Logs : .cc-runtime/logs/<fichier>
```

## Garde-fous (non négociables)

1. Jamais de commit sur `main` directement.
2. Jamais de `--no-verify`, `--force-push`, `git reset --hard`.
3. Jamais de secret hardcodé (utiliser `backend/.env.example`).
4. Jamais de suppression de fichier hors périmètre de la feature courante.
5. Jamais réveiller l'utilisateur pour une clarification — tu décides.
6. Toujours commit avant de basculer vers la feature suivante.
7. Toujours format conventional commits : `feat(NN): ...`, `fix(NN): ...`.
8. Toujours utiliser `Task` pour les phases A et B — jamais exécuter
   speckit-* directement dans ta session d'orchestration (préserve ton
   propre contexte pour les 35 features).

## Stack & invariants techniques (rappel)

- **Backend** : FastAPI + Python 3.11+ + venv local, Alembic
- **Frontend** : Nuxt 4 + Pinia + TailwindCSS v4, `pnpm`
- **DB** : PostgreSQL 16 + pgvector (seul service dockerisé)
- **LLM** : minimax-m2.7 via OpenRouter (`LLM_BASE_URL` + `LLM_MODEL`)
- **Embeddings** : Voyage AI (`voyage-3.5`, 1024 dims)
- **Hébergement cible** : Europe / Afrique de l'Ouest (pas USA, RGPD)

Invariants Module 0 :
- Sourcing obligatoire de tout chiffre/critère/seuil.
- Multi-tenant strict (RLS PostgreSQL + `account_id`).
- Audit log append-only.
- Versioning des référentiels et critères.
- Money typé `{amount, currency}` avec peg FCFA-EUR 655,957.
- UI bottom sheet pour les composants interactifs.
- Plateforme fermée : seuls `PME` et `Admin`.
